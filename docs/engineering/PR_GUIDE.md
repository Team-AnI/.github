# PR Guide (A&I) — PR 작성자용

이 문서는 A&I에서 Pull Request(PR)를 **쉽고 일관되게** 작성하기 위한 가이드입니다.  
처음 오신 분은 이 문서대로만 작성해도 충분합니다.

---

## 0) PR이 중요한 이유

PR은 “코드 제출”이 아니라,

- 내가 한 변경을 **다른 사람이 이해**할 수 있게 공유하고
- 함께 **품질을 올린 뒤**
- **안전하게 합치는 과정**입니다.

---

## 1) PR 만들기 전 체크 (1분)

- [ ] PR 목적이 하나인가? (기능 1개 / 버그 1개 / 문서 1개)
- [ ] 변경 범위가 너무 크지 않은가?
  - 크다면 PR을 쪼개는 게 더 빠릅니다.

---

## 2) 브랜치 네이밍 (권장)

- `feature/<topic>` : 기능
- `fix/<bug>` : 버그
- `docs/<topic>` : 문서
- `refactor/<topic>` : 리팩토링

예)
- `feature/login`
- `docs/onboarding`

---

## 3) PR 제목 규칙 (권장)

커밋 컨벤션의 Type을 PR 제목에도 맞추면 검색/정리가 쉬워집니다.

예)
- `feat(auth): add login flow`
- `fix(api): handle null response`
- `docs(readme): update onboarding`

---

## 4) PR 본문은 “딱 2줄”만 지켜도 합격 ✅

PR을 올릴 때 아래 두 줄만 꼭 적어주세요.

### ✅ 변경 내용 (What)
- 무엇을 바꿨는지 한 줄 요약

### ✅ 확인 방법 (How to check)
- “내가 어떻게 확인했는지”를 **쉬운 말로** 적기
- 실행을 못 했으면 “못 했다”고 적어도 됩니다 (대신 이유/대안을 적기)

#### 예시 — 문서 변경
- 변경 내용: README에 실행 방법 추가
- 확인 방법: 문서 변경이라 실행 없음 / 링크 동작 확인

#### 예시 — 프론트/모바일
- 변경 내용: 로그인 화면 UI 및 상태 처리 수정
- 확인 방법: 에뮬레이터에서 로그인/로그아웃 시나리오 확인

#### 예시 — 백엔드
- 변경 내용: 사용자 조회 API 예외 처리 추가
- 확인 방법: 로컬에서 API 호출 테스트 / 기존 테스트 통과

#### 예시 — AI/노트북
- 변경 내용: 전처리 파이프라인 개선
- 확인 방법: 샘플 데이터로 노트북 재실행 / 결과 스냅샷 첨부

---

## 5) (권장) 첨부하면 리뷰가 빨라져요

가능하면 아래 중 하나라도 붙여주세요.

- UI 변경: 스크린샷/짧은 영상
- API 변경: 요청/응답 예시
- 버그 수정: 재현 조건 + 수정 후 결과

---

## 6) PR 올린 뒤(리뷰 받는 법)

- 리뷰 코멘트가 달리면:
  1) 코멘트 반영해서 커밋/푸시
  2) “반영 완료!” 한 줄 댓글(선택)
- 논쟁이 생기면:
  - “취향”인지 “버그/설계”인지 먼저 분리하고,
  - 팀 룰(문서) 기준으로 합의합니다.

---

## 7) PR 크기 기준(권장)

- PR이 너무 크면 리뷰가 느려집니다.
- 기준 예시:
  - 문서/리팩토링: 가능하면 작게
  - 기능 추가: 기능 단위로 나눠서
